Git Workflow That Works
One branch per feature/fix
Keep main clean ‚Äî all work happens on branches
Small, frequent commits
Easier to revert, easier to review
Descriptive commit messages
Format: "feat:", "fix:", "chore:" ‚Äî makes history scannable
Push regularly
Don't lose work ‚Äî push after significant changes
# Standard workflow
git checkout -b fix/issue-123
# ... make changes ...
git add . && git commit -m "fix: resolve login bug"
git push -u origin fix/issue-123

PRs & Code Review
‚úÖ Do This
One PR = one concern
Don't bundle unrelated changes
CI must pass before merge
No exceptions ‚Äî green builds only
Link to issue in PR description
"Fixes #123" auto-closes the issue
‚ùå Avoid
Merging your own PRs
Always get a second pair of eyes
Giant PRs (500+ lines)
Split into smaller, reviewable chunks
Pushing directly to main
Branch protection exists for a reason

Testing Strategy
üéØ
Test the critical path first
Navigation, core actions, data display ‚Äî if these break, users notice immediately.
üîÑ
E2E tests catch what unit tests miss
CSS changes breaking selectors, z-index issues, timing bugs ‚Äî they show up in E2E.
üì∏
Save debug artifacts on failure
Screenshots, HTML snapshots, console logs ‚Äî you'll need them when CI fails.
# Run E2E tests locally before pushing
npm run test:e2e

# Run specific test
npm run test:e2e -- --grep "Navigation"

Hard Lessons
üé®
Never change HTML structure + CSS together
JS selectors break silently. Visual redesigns should be CSS-only when possible.
üìÅ
One source of truth for code
We lost 8 hours of work with dual workspaces. Pick one location, stick to it.

Small commits = easy reverts
When a feature breaks production, reverting one commit is better than untangling 20.
